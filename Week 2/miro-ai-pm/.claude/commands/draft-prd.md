Read the following files before drafting:
- product-context.md
- goals.md
- current-projects.md
- Anything in research/ that mentions the feature I name below (if that folder exists)

Ask me one question: "What feature, problem, or decision should this PRD cover?"
Wait for my answer before writing anything.

Then draft a PRD with this structure:

## Context
Why we're considering this now. What's happening in the product, the market, Miro AI usage data, or enterprise conversations that makes this relevant. One or two short paragraphs.

## Problem
The user problem, stated concretely. Who has it (use the primary users defined in product-context.md). When they have it. What they do today instead.

## Proposal
What we'd build. One or two paragraphs. No feature list yet. Name the trade-off explicitly: what we'd stop doing or deprioritise to ship this.

## Success Criteria
How we'll know this worked. Specific metrics with deadlines, tied to a Q2 goal from goals.md where possible. Not "users love it".

## Open Questions
What we don't know yet. One sentence per question. Things to resolve before building or during early discovery.

## Out of Scope
What this PRD is not about. State it so reviewers don't spend time there.

Length: aim for one page. 400 to 600 words. If a section needs more, fine. If it needs less, don't pad.

Save the draft to `prds/[short-feature-name]-[today's date YYYY-MM-DD].md`. If the `prds/` folder doesn't exist, create it first.
